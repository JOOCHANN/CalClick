# v0.2-a — Social Foundation

## 한 줄 목표
"이 사람 누구지?" → 닉네임 눌러서 공개 프로필로 이동 · 팔로우 가능.

## Gate
- `/u/[nickname]` 접속 시 공개 프로필 렌더 (닉네임 · 아바타 · 총 기록일 · 스트릭 · 팔로워/팔로잉 수)
- 팔로우/언팔로우 버튼 작동 · 팔로워 카운트 반영
- 팔로워/팔로잉 목록 페이지 (`/u/[nickname]/followers`, `/following`)
- `/me`에서 "내 프로필 보기" 버튼으로 공개 프로필 이동

## 작업 단계

### step1 — 스키마 (마이그레이션 0009)
- `profiles`에 컬럼 추가:
  - `avatar_emoji text check (avatar_emoji in ('🍚','🍜','🥗','🍳','🍎','🍱'))` 기본 `'🍚'` — 프리셋만 허용(ZWJ 이모지 렌더 깨짐 방지)
  - `bio text check (char_length(bio) <= 80)`
- `follows` 테이블:
  ```sql
  create table follows (
    follower_id uuid references auth.users on delete cascade,
    followee_id uuid references auth.users on delete cascade,
    created_at timestamptz default now(),
    primary key (follower_id, followee_id),
    check (follower_id <> followee_id)
  );
  create index follows_followee_idx on follows(followee_id);
  ```
- `follows` RLS — 읽기 전체 허용, 쓰기는 본인 follower_id만:
  ```sql
  alter table follows enable row level security;
  create policy "follows read" on follows for select using (true);
  create policy "follow self" on follows for insert with check (follower_id = auth.uid());
  create policy "unfollow self" on follows for delete using (follower_id = auth.uid());
  ```
- **`profiles` RLS는 풀지 않는다** (민감정보 유출 방어선). 원본은 본인만 read/write 유지. 공개 노출은 별도 view로 분리:
  ```sql
  -- profiles는 원래대로 own-only 유지 (0001 정책 그대로)

  create view profiles_public with (security_invoker = true) as
    select id, nickname, avatar_emoji, bio
    from profiles
    where nickname is not null;

  -- view는 RLS를 원 테이블에서 상속받으므로, 공개 접근은 security definer 함수로 래핑:
  create or replace function public_profile(p_nickname text)
  returns table (id uuid, nickname text, avatar_emoji text, bio text)
  language sql security definer stable set search_path = public as $$
    select id, nickname, avatar_emoji, bio
    from profiles
    where lower(nickname) = lower(p_nickname);
  $$;
  grant execute on function public_profile(text) to anon, authenticated;
  ```
  → 클라이언트가 Supabase JS로 `profiles.*` 직접 조회해도 `current_weight_kg`·`goal_kcal`·`birth_year`·`sex`·`height_cm` 접근 불가. 건강정보 보호(개인정보보호법상 민감정보)가 RLS 수준에서 차단됨. 공개 조회는 `rpc('public_profile', …)` 또는 서버 API 경유.

### step2 — API
- `GET /api/u/[nickname]` — 공개 프로필 (id, nickname, avatar_emoji, bio, 총 기록일, 현재 스트릭, 팔로워수, 팔로잉수, 내가 팔로우 중인지)
- `POST /api/follow` body `{ followee_id }` → insert
- `DELETE /api/follow` body `{ followee_id }` → delete
- `GET /api/u/[nickname]/followers?cursor=` — 각 항목 `{ id, nickname, avatar_emoji, viewer_is_following }` (내 follows와 join해 맞팔 버튼용 상태 포함)
- `GET /api/u/[nickname]/following?cursor=` — 동일 필드

> 닉네임은 변경 가능(1일 1회)하므로 follows/피드 join은 항상 `user_id` 기반, 닉네임 캐시 금지. URL 닉네임은 매 요청마다 `public_profile()`로 resolve.

### step3 — UI
- `/u/[nickname]/page.tsx` (Server Component): 헤더(아바타·닉네임·bio) + stats 카드 3개 + 팔로우 버튼 + 팔로우 목록 링크
- `/u/[nickname]/followers/page.tsx`, `/following/page.tsx`: 리스트 + 무한스크롤 (cursor)
- `/me` 상단에 "내 프로필 공개 보기" 버튼 (닉네임 없으면 설정 유도)
- `/settings` 에 avatar_emoji 선택(6개 프리셋: 🍚 🍜 🥗 🍳 🍎 🍱) + bio 입력

### step4 — 설정 페이지 반영 & 규칙
- bio 80자 제한 validator (zod)
- 닉네임 없는 유저 접근 시 `/settings/nickname`로 리다이렉트 (SEO 상 404 대신)
- 자기 자신 팔로우 불가 (DB check + API 에러)

## 자기 검토 (plan v2 — v1 리뷰 반영)
1. **(해결) profiles RLS 전체 공개의 위험** → v1에서 API 화이트리스트로 막으려 했으나 Supabase JS가 브라우저에서 직접 테이블에 붙는 구조라 API 우회 가능. v2에서 `profiles` 본 테이블은 own-only 유지하고, 공개 필드만 `public_profile()` security definer 함수로 노출. 민감정보가 RLS 수준에서 차단됨.
2. **(해결) 팔로워 리스트에 맞팔 상태 미포함** → v1은 닉네임·아바타만 반환해 "팔로우" 버튼 상태를 클라에서 재조회해야 했음. v2에서 `viewer_is_following` 포함(페이지당 1회 join).
3. **(해결) avatar_emoji 자유 텍스트 위험** → v1은 `text`에 길이 1 가정. ZWJ 시퀀스(👨‍👩‍👧)로 우회/깨짐 가능. v2에서 프리셋 6개만 허용하는 check constraint.
4. **(유지) 닉네임 변경으로 공개 URL 유실** → 1일 1회 제한 + `/u/old-nick` 404. 리다이렉트 테이블 over-engineering. follows/피드 join은 user_id 기반으로 통일해 닉네임 캐시 이슈 회피.
5. **(유지) 팔로우 카운트 쿼리 비용** → 직접 `count(*)` 집계. 10만+ 규모 시 `profile_counts` 비정규화로 이전. 지금 불필요.
